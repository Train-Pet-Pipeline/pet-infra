### 1.3 不变量（整个 rental 期间硬约束，破坏 = bug）

1. **零硬编码**：所有数值从 `params.yaml` 读取。
2. **零绕过**：发现问题不能 try/except 静默或 patch 跳过；必须找根因。
3. **零跳过 dedup**：CLAUDE.md 明令。
4. **零修改已提交 Alembic**：只新增。
5. **数据库只走 store.py**：不直连。
6. **修复后强制 retest**：fix → 重装 → 重跑该步骤 → 通过才能继续；retest 证据写入 finding。
7. **所有 PR 走 feature/* → dev → main**：rental 上 fix 的 commit 也走标准分支策略。
8. **W&B 残留 zero-tolerance**：no-wandb-residue.yml 必须在 fix PR 上绿。
9. **北极星 / 生态指标违规 = 结构性问题**：
   - 北极星 4 维（PHASE_DOD_TEMPLATE §5）任一维 < 3：
     - **Pluggability** — 新模型/新 modality 是否只加 plugin 不改核心？
     - **Flexibility** — 配置能否纯 Hydra override 不改代码？
     - **Extensibility** — 新增 registry 成员是否无需改 orchestrator？
     - **Comparability** — 同一 recipe 切不同 plugin 指标同格式？
   - 生态指标违规：跨仓版本不在 matrix 锁定行 / PR 跳 reviewer 或 CI gate / 文档同步缺失
   - 处理：**走 §4.3 结构性 fix 协议**（正式 PR + 必要时跨仓 refactor + matrix 行 bump + tag），整链 retest

## §4 Phase 3 + Find→Fix→Retest 闭环 + 决策合成协议

### 4.1 Phase 3 跨切关注（Cross-Cutting）

Phase 0/1/2 都按仓 / 按 step 走，Phase 3 验证**跨仓组合**真的解耦。

| # | 跨切场景 | 验证 | Pass 判据 | 北极星维 |
|---|---|---|---|---|
| X1 | recipe 切 evaluator plugin 不改代码（与 E3 同源但加入 override 维度）| 复用 E3 输出 + 额外手动 `pet run cross_modal_fusion_eval evaluator.weights='[0.7, 0.3]'` Hydra override 改 weighted plugin 权重 | E3 三 plugin output schema 一致；override 路径生效（report 中权重数值与 override 匹配） | **Comparability + Flexibility** |
| X2 | 多 modality 端到端 | 同次 `pet run` 跑 vision + audio + fusion eval（一份 recipe 三 stage） | 三份 metric 同 root；fusion 引用前两份 | **Pluggability + Extensibility** |
| X3 | replay 跨 commit 漂移检测 | Phase 1.7 的 ModelCard 在另一 commit 上 replay | 检测到 git drift warning（不报错，只警告） | Comparability |
| X4 | variations launcher 多 plugin 比较 | 一份 recipe variations: `[trainer_a, trainer_b]` × `[evaluator_a, evaluator_b]` | 4 个 run 自动跑；report 可并排比较 | **Pluggability + Comparability** |
| X5 | sweep + ClearML 集成 | 一个 2x2 sweep + ClearML offline mode | sweep 4 个 run 全打到 ClearML offline；UI/log 可见 | Flexibility |

| X7 | storage 后端切换不改 plugin | 同 trainer recipe 改 `storage: local` → `s3` | trainer 不感知，artifact 落对地方 | **Pluggability** |
| X8 | gate 拦截 OTA 发布 | gate fail 时，OtaStageRunner 应拒绝发布 | release 不发出；状态机回 `gate_failed` | Flexibility |

任一 X 项被打回 ≠ Pass = 北极星该维度 < 3 = 结构性问题。

### 4.2 Find → Fix → Retest 闭环

```
发现异常
   │
   ▼
[第 1 步 stop] 停下当前 step，不继续往下走
   │
   ▼
[第 2 步 evidence] 收集证据（log / stack / 复现命令 / 输入 hash）
   │
   ▼
[第 3 步 root cause] 调 systematic-debugging skill；不接受"应该是 X" 必须证据链
   │
   ▼
[第 4 步 severity] 分级（STRUCTURAL / CRITICAL / HIGH / MEDIUM / LOW）
   │
   ├─ STRUCTURAL → 走 §4.3 结构性 fix 协议
   ├─ CRITICAL/HIGH → 走 §4.4 立即 fix（A 路径）
   └─ MEDIUM/LOW → 走 §4.5 攒批量（B 路径）
   │
   ▼
[第 5 步 fix] 写 fix；调 test-driven-development skill 加回归测
   │
   ▼
[第 6 步 retest] 重装 → 重跑该 step → 通过 → 同时跑该 phase 已通过 step 的 smoke 验无副作用
   │
   ▼
[第 7 步 doc] finding md 补 fix commit / PR / retest 截图
   │
   ▼
[第 8 步 verify] 调 verification-before-completion skill 不口头声称 pass
   │
   ▼
关闭 finding，继续 Phase
```

**关键约束**：
- 第 3 步禁止跳：30 分钟还找不到根因 → 调 critic agent 二意见 → 还不行写 finding 标 `BLOCKED` + 决策"是降级、跳过、还是退租"。
- 第 6 步**必须 retest**，不能"我觉得 fix 了"就关闭。
- 第 8 步**必须有命令输出截图/log** 作为 evidence，不接受"看起来对"。

### 4.3 结构性 fix 协议（北极星 / 生态指标违规）

| 步 | 操作 |
|---|---|
| 1 | finding 标 `severity: STRUCTURAL`，明确违反哪一维 |
| 2 | 评估影响面：单仓内重构？跨仓 API 变更？schema 变更？ |
| 3 | 单仓内 → feature/* PR + reviewer + tag minor/major bump |
| 4 | 跨仓 → 决策树：先改 schema/infra → tag → 下游按依赖链 PR；matrix 行 bump |
| 5 | 同步 DEV_GUIDE 对应章节 + matrix_history 追加条目 |
| 6 | rental 上重装受影响仓的新 tag → **全链 retest**（Phase 0+1+2 受影响 step） |
| 7 | 写 retro 段落分析为何之前没暴露（CI gap？测试 gap？） |

### 4.4 立即 fix（CRITICAL / HIGH，A 路径）

简化版结构性流程：feature/* PR → CI → quick approve → merge dev → rental 上 git pull dev → editable retest。

Tag/release 推到 rental 结束后批量。

**Reviewer 缺位时**（用户睡觉等）：CRITICAL/HIGH PR 写好但**不 self-merge**；rental 上用本地 fix branch + editable install 先继续推进 + retest（事实上等同 B 路径），PR 等用户醒后 review。
此机制下 STRUCTURAL fix 例外：必须用户 review，stop 等。

### 4.5 攒批量 fix（MEDIUM / LOW，B 路径）

rental 上：
- 本地 fix branch（`fix/eco-validation-batch-N`）累积小 commit
- 每 commit 都对应一个 finding
- 当前 step 只 editable install + retest，不开 PR
- rental 结束统一推 + 按主题分组拆 PR

### 4.6 决策合成协议

Rental 期间任何分叉点，按下面四维**复合判断**（参考性框架，不是机械算法）：

| 维度 | 评估问句 |
|---|---|
| **测试精神** | 这个选项是否破坏 zero-bypass / 必 retest / 证据先于声明？破坏 = 否决 |
| **北极星 4 维** | 这个选项是否让 pluggability/flexibility/extensibility/comparability 任一维降分？大幅降分 = 否决 |
| **生态精神** | 是否违反 peer-dep / 单一来源 / no-hardcode / 结构化日志 / retry-over-silent / BSL license？违反 = 否决 |
| **工程最佳实践** | SOLID / YAGNI / 根因优先 / test pyramid / 不引入兼容层？违反 = 重新设计 |

**评估流程**（极简、判断为主）：
1. 列出至少 2 个候选（不允许"只有一种走法"思维）
2. 对每候选检查 4 条硬否决：① 破坏 zero-bypass / 必 retest？② 让北极星某维大幅降分？③ 违反生态精神（peer-dep / 单一来源 / no-hardcode / structured log / retry）？④ 严重违反工程最佳实践？
3. **若选项是 "defer / 推迟修"**，多走一道 self-check（F008 retro 加）：**"Is this defer truly hardware-blocked, or is it a software gap testable on any GPU?"** 软件 pluggability 缺陷不可借 hardware phase 之名 defer。
4. 任一条触发否决 → 该选项淘汰；剩余选项按"更快、更简单"挑
5. **决策日志：1 句话理由写入 report.md**

明显场景（如 OOM 改 batch size）不必走完整流程，直接做 + 1 句记录即可。

**示例**（rental 上真实可能遇到）：
> 场景：T2（Qwen2-VL-2B 全参 SFT）OOM。
> 候选 A：换 batch_size=1 + grad_accum=8 → 测试精神 0 / 北极星 0 / 生态 0 / 工程 0 = 0（保住测试）
> 候选 B：换更小模型（Qwen2-VL-1B） → 测试精神 -1（绕过：未验证 2B 全参就走，不算测出能力）= 否决
> 候选 C：跑 LoRA 替代 → 测试精神 -1（与 T1 重复，没验证全参路径）= 否决
> **选 A**，记录"显存策略调整不绕过测试目的"

### 4.7 必须停下问的事项（自主权边界）

自主推进，但下列两类必须停 + 通知用户（即使授权了 Phase 自主）：

1. **钱**：Sonnet 调用估算 > $20，或要切 Doubao；rental 续租超原预算 50%
2. **跨仓破坏性变更**：影响生态版本契约的 STRUCTURAL fix（≥ 3 仓连锁 PR / 改 pet-schema 公开数据模型）

其它一律自主推进，rental 结束 retro 一次性汇报。

### 5.3 Stop Criteria（rental 自然结束的条件）

按优先级，任一触发即收尾：

1. **Phase 3 全部 ✅** + **北极星 4 维 ≥ 3** + **生态指标全 ✅** + **所有 STRUCTURAL/CRITICAL finding 关闭** → **目标达成，自然 stop**
2. 时间预算耗尽（如 24h）且 Phase 2 ≥ 6/9 仓子矩阵走完（含 pet-train + pet-eval + pet-infra 三个核心仓必走完） → **stop + 写 retro 标"局部完成"**
3. STRUCTURAL finding 链需要 ≥ 3 仓重构 → **stop**（按 §4.7 通知用户）
4. CUDA / 环境根本性问题 30 分钟内无解 → **stop + fail-fast 退租**
5. 用户直接 stop（用户可随时中断）

**反模式（这些情况要继续推进）**：
- 单个 finding 难修：调 critic / debugger 二次意见，最多 90min 后降级 deferred 继续
- 某仓矩阵某项 fail：写 finding 后**继续走该仓其他 capability**（不要全仓阻塞）
- 时间紧但 Phase 3 没开始：保留 Phase 3 至少 X1 / X2 / X7 三项最关键的，砍其他
